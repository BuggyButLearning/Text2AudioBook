<objective>
## Goal
Harden the Ollama local-provider path: explicit connectivity check (`ollama_reachable(base_url)`), clearer pre-flight error when Ollama is not running, and a sharper "no TTS endpoint" error message that points the user toward Kokoro (the v0.1 local fallback). Document the limitation in module docstrings + PROJECT.md decision log.

## Purpose
Phase 2.1 shipped Ollama model discovery (curated allowlist filter, EMPTY-with-reason). Phase 2 shipped the synthesis-side `_validate_ollama_model_support` guard. What's missing is the pre-flight connectivity check that distinguishes "Ollama not running" from "Ollama running but no TTS model" — both currently surface as EMPTY but the user-facing fix differs (start `ollama serve` vs `ollama pull bark`).

## Output
- New `ollama_reachable(base_url, timeout=2.0, request_fn=None) -> tuple[bool, str | None]` helper in `model_discovery.py`. Pure-Python with `request_fn` injection seam.
- Refined Ollama-not-running error in `_discover_ollama`: error message hints to start `ollama serve`.
- Refined Ollama-no-TTS-endpoint error in `convert_text_chunk_to_speech`: mentions Kokoro as the local alternative (Phase 6.2).
- Tests: ≥ 8 new tests (connectivity probe, refined error messages, Kokoro hint).
</objective>

<acceptance_criteria>

## AC-1: `ollama_reachable` connectivity helper
Pure-Python. Hits `{base_url}/api/version` (a known cheap endpoint). Returns `(True, None)` on 2xx response. Returns `(False, <reason>)` on connection refused / timeout / non-2xx. `request_fn` injection seam for tests.

## AC-2: Sharper "Ollama not running" hint
`_discover_ollama`'s connection-refused branch returns `DiscoveryResult.error` containing both the raw cause AND a hint ("Is `ollama serve` running?"). Existing tests for `_discover_ollama` continue to pass (assert "contains" not "equals").

## AC-3: Sharper "no TTS endpoint" hint
`convert_text_chunk_to_speech` Ollama branch error message mentions Kokoro: "...not available through standard Ollama endpoints in this build. For local synthesis, use the Kokoro provider (Phase 6.2 once landed)."

## AC-4: Tests
- `TestOllamaReachable`: 200 → (True, None); connection refused → (False, contains "refused"); timeout → (False, contains "timed out"); 500 → (False, contains "500")
- Update existing Ollama discovery error tests to assert "contains" rather than exact match
- New test in `test_tts_conversion.py` asserting the Kokoro-hint string appears in the no-endpoint error

## AC-5: Regression green
≥ 251 tests (243 + ≥ 8 new), exit 0, < 5.0s. `git diff providers.py settings.py text_processing.py main.py combine_and_convert.py` = no changes.
