## Goal
Wire `tts_conversion.py` to consume the immutable `providers.PROVIDER_REGISTRY` from Phase 1 as its single source of truth for provider capabilities (voices, model patterns, fallback models, output formats, concurrency defaults). Replace inline duplications (`_validate_ollama_model_support` token list; `_filter_openai_tts_models` naive "tts" substring) with registry-driven lookups. Migrate the OpenAI SDK call from the deprecated `response.stream_to_file(...)` to the recommended `client.audio.speech.with_streaming_response.create(...)` context-manager pattern. Add structured chunk-level logging. Keep the existing retry-with-backoff + bounded-concurrency behavior; verify and characterize.

## Purpose
Phase 1 shipped the contract layer (`providers.py`); the pre-PAUL commit `f2ff098` already shipped the SDK migration baseline (`_write_openai_speech` via `client.audio.speech.create`). Phase 2's remaining work is to **close the gap between contract and implementation** — `tts_conversion.py` currently restates capability logic that already lives in the registry. Letting two copies drift is exactly the failure mode Phase 1's M2 audit finding addressed for HF revisions; the same hygiene applies here.

Secondarily, `response.stream_to_file()` emits a `DeprecationWarning` in openai 2.x and will be removed in a future major. The non-deprecated `with_streaming_response.create(...)` context-manager API is the official replacement. Phase 2 is the right time to make the switch since this code is being touched anyway.

The 145-test regression net stays green; behavior shifts get characterized with `# CHARACTERIZED — Phase 2 change` comments.
