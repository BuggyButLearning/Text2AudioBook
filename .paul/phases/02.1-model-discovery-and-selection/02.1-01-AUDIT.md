## 1. Executive Verdict

Phase 2.1 plan is well-scoped: it correctly extracts discovery from synthesis, respects Phase 1's immutability invariants, characterizes the new Ollama allowlist filter as a deliberate behavior shift, and re-uses the Phase 1 / Phase 2 monkeypatch + module-level-regex patterns instead of reinventing them. But the pre-audit version had four classes of enterprise-grade defect:

1. **Credential leakage surface.** New logging path in `_discover_openai` exception branch passes `exc` to `logging.warning(..., exc)`. OpenAI SDK error messages can include request payload fragments. Phase 2 explicitly tested `"sk-SECRET-KEY-123" not in joined_log` for the synthesis path; Phase 2.1 introduces an analogous logging surface and did NOT propagate that invariant. Same defect class as the Phase 2 audit's S2-style failures.
2. **Source-enum semantic inconsistency.** Plan's `_discover_openai` collapses "API exception → FALLBACK" and "API up + allowlist filtered everything → FALLBACK" into the same source value. But Ollama (AC-3) correctly distinguishes "HTTP error → EMPTY" from "HTTP OK but allowlist filtered everything → EMPTY with reason". OpenAI must match for callers to reason consistently.
3. **Cache-key drift risk.** Plan uses raw `api_key` as cache identity (Phase 2.1 G2) — both a security smell (raw key in module-level dict) and a UX bug (calling `discover_models("Ollama", ollama_base_url=None)` then `discover_models("Ollama", ollama_base_url="http://localhost:11434")` creates two cache entries for the same canonical URL). Normalize identity before keying.
4. **Cached-error stickiness undocumented.** Pre-audit AC-4 says "cache entries do NOT expire on time" — true, but the plan never makes explicit that this means FALLBACK and EMPTY responses are cached just like LIVE. A transient OpenAI outage at app start would burn into the cache until the user happened to hit Refresh. That's an intentional design choice for a desktop app (predictable, no surprise re-fetches), but it needs to be a documented contract with an explicit test, not an emergent property.

After applying the must-have + strongly-recommended upgrades inline, the plan is approved for APPLY.

Pre-upgrade: would not sign for production. Credential-leak in logs is non-negotiable for a system that handles user-supplied API keys.
Post-upgrade: yes.

## 2. What Is Solid (Do Not Change)

- **Boundary discipline.** `providers.py` untouched. `_filter_openai_tts_models`, `_validate_ollama_model_support`, and the Phase 2 synthesis helpers stay where Phase 2 put them. No leak into Phase 4 (GUI) or Phase 6 (Ollama) scope.
- **Behavior-shift pre-characterization.** Plan anticipates the `test_skips_malformed_entries` assertion change (`["ok"]` → `[]`) and pre-assigns a `# CHARACTERIZED — Phase 2.1` comment. Adds a companion positive-case test.
- **Back-compat surface design.** Thin shims in `tts_conversion.py` preserve `main.py`'s import path. Zero risk of GUI regression in a phase that doesn't own the GUI. `list_available_models` keeps `list[str]` return type so existing 5 tests + main.py see no signature change.
- **Module-level compiled regex** (Phase 2 audit S1 pattern). Cheap per-call, fail-fast at module import if registry ever shipped a malformed pattern.
- **Frozen DiscoveryResult.** `@dataclass(frozen=True)` + tuple-not-list for models + an explicit mutation-attempt test. Callers can't accidentally corrupt a cached result. Matches Phase 1's ProviderCapability immutability pattern.
- **Explicit cache invalidation entry point.** `invalidate_cache(provider=None)` is the public hook Phase 4's "Refresh Models" button needs. No time-based TTL — discovery is best-effort and refresh is user-initiated. Right calibration for a desktop GUI.
- **Cache cleared between tests via autouse fixture.** Shared module state would otherwise pollute test isolation; the autouse `invalidate_cache()` before+after each test is the correct hygiene.
- **Curated allowlist sourced from the registry.** PRD §14.1(a) live: a user with `llama3` installed won't see it in the TTS dropdown. The pattern is the registry's, not a hardcoded list in discovery — drift impossible.

## 3. Enterprise Gaps Identified

### G1 — `api_key` value can leak through error log
**Pre-audit:** `_discover_openai` calls `logging.warning("OpenAI discovery failed; using registry fallback: %s", exc)`. The `exc` object's string representation for openai SDK errors can include URL-encoded headers or request bodies that contain the Bearer token in some failure modes. Phase 2's TestChunkLogging explicitly tested `"sk-SECRET-KEY-123" not in joined_log` for the synthesis path. Phase 2.1 introduces a parallel logging surface with the same risk and did not propagate the invariant.

Severity: **must-have**. Credential-in-log is a release-blocking compliance issue.

### G2 — OpenAI Source-enum inconsistent with Ollama
**Pre-audit:** `_discover_openai` returns `Source.FALLBACK` in two distinct scenarios:
  - Network/auth/SDK exception → FALLBACK with error
  - API responded with 200 OK but allowlist filtered everything → FALLBACK with "no models matched allowlist"

Ollama (AC-3) correctly distinguishes:
  - HTTP exception → EMPTY with error
  - HTTP OK + allowlist empty → EMPTY with "no models matched registry allowlist"

A Phase 4 UI consumer trying to decide messaging (red banner for "OpenAI is down" vs yellow banner for "your OpenAI account has no TTS models enabled") cannot distinguish these from FALLBACK alone. The semantics should be: **FALLBACK = an exception happened; EMPTY-with-error = upstream responded but yielded nothing useful**.

Severity: **strongly recommended** (audit-trail integrity + UX hygiene).

### G3 — Ollama cache identity is not canonical
**Pre-audit:** Cache key is `(provider, ollama_base_url)` where `ollama_base_url` is the raw arg. `discover_models("Ollama", ollama_base_url=None)` and `discover_models("Ollama", ollama_base_url="http://localhost:11434/")` hit the same `/api/tags` endpoint (the function defaults None → "http://localhost:11434" and rstrips trailing slash) but produce **three separate cache entries** because the raw values differ. Wasted cache hits + multiple network calls for the same canonical URL.

Severity: **strongly recommended**.

### G4 — Cache stickiness on transient failures undocumented
**Pre-audit:** AC-4 says "cache entries do NOT expire on time — only on explicit invalidate". True, but never makes explicit that this means a one-time network blip permanently poisons the cache with FALLBACK/EMPTY until the user hits Refresh. For a desktop app this is **acceptable design** (predictable, no surprise refresh storms), but it is also **an audit gap**: the contract isn't tested. If a future Phase changes this to "only cache LIVE results" the rest of the codebase has no guard.

Severity: **strongly recommended** (contract documentation + characterization test).

### G5 — `None`-vs-default `api_key` cache separation untested
**Pre-audit:** `_CACHE` uses `(provider, identity)` where identity is `api_key` for OpenAI. `discover_models("OpenAI", api_key=None)` and `discover_models("OpenAI")` (default api_key=None) should hit the same cache entry. `discover_models("OpenAI", api_key="sk-x")` should hit a different one. The "different api keys" test covers `"sk-A"` vs `"sk-B"` but not `None` vs `"sk-x"`. Edge case worth pinning.

Severity: **strongly recommended**.

### G6 — `error` field is a user-facing surface but unspecified
**Pre-audit:** AC-1 declares `error: str | None = None` but never states that this is the field Phase 4 GUI will surface to the user. Two implications:
  - Field must be a short, user-readable string (not a stack trace or repr)
  - Field must never contain credentials (covered by G1 / M1)

Severity: **strongly recommended** (UX contract clarification).

### G7 — Unbounded cache growth on rotating API keys
**Pre-audit:** Cache is unbounded. Pathological case: an automated test or a key-rotation workflow calls `discover_models("OpenAI", api_key=…)` with N distinct keys, growing `_CACHE` to N entries. Not realistic for a single-user desktop GUI but technically a leak.

Severity: **can safely defer** — desktop app with single user, single key. Phase 4 GUI lifecycle naturally bounds it.

### G8 — Thread-safety of `_CACHE`
**Pre-audit:** `_CACHE` is a plain dict. Python dict ops are mostly atomic but read-then-write races exist if `convert_text_to_speech` (which uses ThreadPoolExecutor) ever invokes `discover_models` from worker threads. Synthesis currently does NOT call discovery from workers (discovery is called from main.py GUI thread before the ThreadPoolExecutor opens), but the contract isn't enforced.

Severity: **can safely defer** — Phase 4 owns GUI threading; if discovery moves into worker threads, that phase adds a `threading.Lock`. v0.1 contract is "discovery is called from main thread only."

### G9 — No exception-type discrimination in OpenAI discovery
**Pre-audit:** `_discover_openai` catches bare `Exception`. AuthenticationError, RateLimitError, APIConnectionError all collapse to FALLBACK. A Phase 4 UI consumer might want to distinguish "your key is bad" from "OpenAI is down" from "you're rate-limited". Same gap class as Phase 2 D1 (retry-everything semantics).

Severity: **can safely defer** — same rationale as Phase 2 D1. Real semantic change deserves its own plan.

## 5. Audit & Compliance Readiness

### Defensible audit evidence
- **Before:** Discovery errors logged via `logging.warning(..., exc)` — `exc` repr potentially contains api_key.
- **After (M1):** `api_key` scrubbed from `str(exc)` before logging or storing in `result.error`. Explicit test proves the invariant.

### Silent failure prevention
- **Before:** OpenAI "API up, allowlist filtered everything" indistinguishable from "API down" in Source-enum. Phase 4 UI would show the same banner for both.
- **After (S1):** `Source.EMPTY` for "live-but-allowlist-filtered" vs `Source.FALLBACK` for "exception". Phase 4 can route messaging correctly.
- **Before:** Cache key drift could send two callers with the same logical URL to different cache entries.
- **After (S2):** Canonical URL normalization at the top of `discover_models` collapses N→1.

### Post-incident reconstruction
- Discovery logs include reported-model count, filtered-model count, and source value. An auditor reading logs can answer "which models did the user see and why" without replaying.
- DiscoveryResult.error is the single user-facing field; documented (S5) and credential-scrubbed (M1).

### Boundary integrity
- Unchanged. `providers.py` untouched. `settings.py` not modified. `main.py` not modified. No new third-party deps. Phase 2 synthesis helpers untouched.

### Test fidelity
- Cache hit/miss verified via call-counter on monkeypatched client (not via timing, which is flaky).
- Provider isolation in cache verified via parallel OpenAI + Ollama counters.
- New (M1) credential-leak test uses `caplog.text` directly to assert the key string never appears anywhere in captured log records.
